# 项目文档生成器

通过六阶段流程生成全面、以用户为中心的项目文档：探测 → 分析 → 发现 → 生成 → 验证 → 精炼。

## 触发条件

- 用户要求"介绍/文档化/分析/详解"某个项目
- 用户要求"理解项目架构/设计/用法"
- 用户要求"为项目写文档"

## 核心原则

> **文档应该映射用户的理解路径，而不是映射代码的模块结构。**

用户理解路径：
```
这是什么？ → 能干什么？ → 有什么亮点？ → 怎么用？ → 怎么深入？ → 怎么贡献？
```

---

## 阶段一：探测（Probe）— 识别项目属性

在深入分析前，先通过特征文件自动识别项目的基本属性。

### 1.1 语言识别

扫描根目录特征文件判断编程语言：

| 特征文件 | → 语言 |
|------|------|
| `go.mod` | Go |
| `package.json` | JavaScript / TypeScript |
| `pyproject.toml` / `setup.py` | Python |
| `Cargo.toml` | Rust |
| `pom.xml` / `build.gradle` | Java / Kotlin |
| `CMakeLists.txt` | C / C++ |
| `Gemfile` | Ruby |
| `composer.json` | PHP |
| `*.csproj` / `*.sln` | C# / .NET |
| `Package.swift` | Swift |
| `pubspec.yaml` | Dart / Flutter |

### 1.2 项目类型识别

根据目录结构和入口文件判断：

| 指征 | → 类型 |
|------|------|
| 有接口/抽象定义但无 `main`/`app` 入口 | 库/框架 |
| 有 `cmd/`、`cli/`、`__main__.py`、bin 目录 | CLI 工具 |
| 有前端框架配置文件 | 前端 Web 应用 |
| 有 `Dockerfile` + `api/`/`proto/`/`controller/` | 微服务/API |
| 有 `packages/` + workspace 配置 | Monorepo |
| 有 `templates/` + `routes/`/`views/` | 全栈/后端 Web 应用 |

### 1.3 分析侧重自适应

| 项目类型 | 重点深入 | 可简略 |
|---------|--------|-------|
| 库/框架 | 核心概念、使用模式、亮点发现、扩展与定制 | 开发指南 |
| CLI 工具 | 使用模式、快速上手、亮点发现 | 扩展与定制、数据流 |
| Web 应用 | 快速上手、使用模式、整体架构 | 技术实现要点 |
| 微服务 | 整体架构、生态与集成、可观测性 | 使用模式 |
| Monorepo | 整体架构、核心概念 | 单包内部细节 |

### 1.4 文档长度与深度自适应

根据项目规模控制文档长度，避免信息过载：

| 项目规模 | 判断标准 | 目标长度 | 要点 |
|---------|--------|--------|------|
| 小型 | < 50 文件，< 5 个核心概念 | ≤ 300 行 | 聚焦核心，每个维度简明描述 |
| 中型 | 50-200 文件 | 500-800 行 | 适当展开亮点和使用模式 |
| 大型 | > 200 文件 | 800-1200 行 | 必须有 TL;DR 概览，亮点和使用模式深入展开 |

**所有文档开头都必须有 100 字以内的"一句话定位 + 核心价值 + 快速跳转目录"**，让读者 10 秒内判断是否值得深读。

### 执行步骤

1. 列出根目录所有文件
2. 匹配特征文件识别语言
3. 检查目录结构识别项目类型
4. 评估项目规模（文件数、目录深度、依赖数量）
5. 根据项目类型确定分析侧重
6. 根据项目规模确定文档目标长度

## 阶段二：分析（Analyze）— 14 个维度

按用户理解阶段分层，逐项收集信息。

### 第一层：建立认知（是什么 & 为什么）

#### 维度 1：项目定位
- **回答的问题**：一句话说清楚这是什么？
- **执行动作**：
  - 读取 `README.md`（及本地化版本如 `README.zh_CN.md`）
  - 读取包描述文件（`doc.go`、`__init__.py`、`package.json` 的 description 等）
  - 提取：项目名称、一句话描述、所属领域、核心价值主张

#### 维度 2：解决的问题与使用场景
- **回答的问题**：什么场景下需要它？
- **执行动作**：
  - 分析 README 的动机/介绍部分
  - 搜索示例目录（`examples/`、`testdata/`）
  - 从示例和文档中识别 3-5 个典型使用场景
  - 提取：痛点分析、典型场景、目标用户群

#### 维度 3：与同类项目的对比
- **回答的问题**：为什么选它而不选别的？
- **执行动作**：
  - 搜索 README 中的对比/替代方案提及
  - 识别同领域的竞争项目
  - 提取核心差异点
  - 必要时使用 WebSearch 验证对比信息
- **客观性约束**（必须遵守）：
  - 对比维度避免非黑即白（✅/❌），使用"强/中/弱"或具体数据描述
  - 不确定的对比项标注"⚠️ 待验证"而非凭印象判断
  - 每个对比项应提供依据来源（如官方文档、benchmark）
  - 承认竞品的优势，不贬低同类项目
  - 对比表应帮助用户做出适合自己的选择，而非证明"本项目最好"

### 第二层：理解设计（怎么设计的）

#### 维度 4：核心理念与设计哲学
- **回答的问题**：它的"灵魂"是什么？
- **执行动作**：
  - 查找设计文档、ADR、RFC
  - 从代码结构中分析模式（接口驱动、约定优于配置等）
  - 识别关键设计决策及其权衡
  - 从 `doc.go`、包注释、架构文档中提取

#### 维度 5：核心概念模型
- **回答的问题**：需要理解哪些关键概念？
- **执行动作**：
  - 识别 5-10 个核心类型/接口/抽象
  - 映射概念间的关系（组合、继承、依赖）
  - 为抽象概念找到现实世界的类比
  - 按语言自适应搜索：
    - Go：`type ... interface`、`type ... struct`
    - Java：`interface ...`、`abstract class ...`
    - Python：`class ...(ABC)`、`class ...(Protocol)`
    - TypeScript：`interface ...`、`type ...`
    - Rust：`trait ...`、`struct ...`

#### 维度 6：整体架构
- **回答的问题**：各部分怎么协作的？
- **执行动作**：
  - 列出顶层目录及职责
  - 分析包/模块间的依赖方向
  - 绘制分层架构图（文本描述）
  - 识别依赖流向：哪层依赖哪层

### 第三层：学会使用（怎么用）

#### 维度 7：快速上手
- **回答的问题**：5 分钟能跑起来吗？
- **执行动作**：
  - 从 README 提取安装步骤
  - 找到最小可运行示例
  - 验证依赖和前置条件
  - 编写清晰的分步快速上手指南

#### 维度 8：核心使用模式
- **回答的问题**：最常见的用法是什么？
- **执行动作**：
  - 从示例、测试、README 中识别 3-5 个典型使用模式
  - 每个模式：提供代码示例 + 解释解决的问题
  - 确保示例引用源码中的实际 API 签名
  - 模式从简单到复杂递进

#### 维度 9：扩展与定制
- **回答的问题**：不满足需求怎么办？
- **执行动作**：
  - 搜索扩展模式：plugin、hook、middleware、callback、factory、registry、strategy
  - 识别所有用户可实现的接口
  - 查找配置选项
  - 用示例说明扩展点

### 第四层：深入掌握（深入理解）

#### 维度 10：数据流与执行流
- **回答的问题**：内部是怎么运转的？
- **执行动作**：
  - 从入口追踪主要执行路径
  - 映射每步的数据变换
  - 识别流式/异步机制（如有）
  - 用文本图描述流转

#### 维度 11：可观测性与调试
- **回答的问题**：出了问题怎么排查？
- **执行动作**：
  - 搜索日志、追踪、指标机制
  - 查找回调/hook/拦截器系统
  - 识别调试工具或开发模式
  - 记录如何添加自定义可观测性

#### 维度 12：生态与集成
- **回答的问题**：如何融入更大的系统？
- **执行动作**：
  - 搜索配套项目（如 `-ext`、`-cli`、`-devops` 后缀）
  - 识别第三方集成
  - 检查插件注册表或扩展目录
  - 记录生态全景

### 第五层：参与贡献

#### 维度 13：开发指南
- **回答的问题**：怎么参与开发？
- **执行动作**：
  - 读取 `CONTRIBUTING.md`（如存在）
  - 读取 CI/CD 配置（`.github/workflows/`、`Jenkinsfile` 等）
  - 识别构建命令、测试命令、lint 命令
  - 记录代码风格要求和 PR 流程

#### 维度 14：技术实现要点
- **回答的问题**：核心模块怎么实现的？
- **执行动作**：
  - 识别技术上有意思的实现细节
  - 记录关键内部机制（序列化、并发、内存管理等）
  - 聚焦非显而易见的实现选择

## 阶段三：发现（Discover）— 挖掘项目亮点

> **这是文档的"灵魂"部分。一个好的项目文档不仅要准确，还要让读者感受到项目的魅力。**

在完成 14 个维度的结构化分析后，以**探索者的新鲜眼光**重新审视项目，挖掘那些让人眼前一亮的特性。

### 发现什么？

#### 3.1 技术亮点

寻找项目中**非显而易见、但一旦理解就觉得精妙**的技术实现：

- **巧妙的抽象**：是否有"一个接口统一多种场景"的设计？
- **优雅的机制**：是否有自动完成复杂工作的内部机制？（如自动流式适配、泛型扇出/扇入）
- **性能优化**：是否有特殊的性能优化手段？
- **类型安全**：是否通过类型系统在编译期消除了整类错误？
- **通用模式**：是否有可复用到其他项目的设计模式？

**执行动作**：
- 阅读核心模块的内部实现代码，而非只看接口
- 搜索 `internal/` 目录下的关键机制
- 对比"用户写法"和"内部实现"的差距，找出框架帮你做了什么
- 关注 `sync`、`generic`、`pool`、`cache` 等关键词

#### 3.2 工程亮点

寻找项目中**体现工程素养**的做法：

- **API 设计**：接口是否简洁、一致、难以误用？
- **向后兼容**：如何处理版本迁移和兼容性？
- **错误信息**：错误提示是否友好、可操作？
- **代码组织**：包结构是否清晰、依赖方向是否合理？
- **文档质量**：代码注释、示例、测试是否覆盖充分？

**执行动作**：
- 阅读错误处理路径，评估错误信息质量
- 检查 `doc.go` 和包注释的完整性
- 查看 `examples/` 和测试覆盖
- 检查是否存在弃用标记（deprecated）和迁移指南

#### 3.3 独特价值

寻找这个项目**区别于同类项目的核心差异**：

- **独创功能**：有没有其他同类项目没有的能力？
- **独特组合**：是否有"1+1>2"的功能组合？
- **生态位**：它填补了什么空白？
- **适用场景**：有没有它特别适合的特定场景？

**执行动作**：
- 回到维度 3 的同类对比，深入分析差异
- 从 README 和 issue 中寻找用户反馈的独特价值
- 搜索项目中其他框架不常有的特性

#### 3.4 "Aha Moment"

寻找那些**让读者恍然大悟**的设计：

- "原来还可以这样用！"
- "这个设计解决了我一直头疼的问题"
- "没想到还能这样实现"
- "这个抽象太优雅了"

**执行动作**：
- 寻找项目中反直觉但正确的设计决策
- 找到"简洁 API 背后复杂实现"的例子
- 识别那些减少用户认知负担的设计

### 输出方式

在文档中以"亮点"专题的形式呈现，每个亮点包含：
1. **一句话概括**：用通俗语言描述这个亮点
2. **为什么值得注意**：解决了什么问题 / 带来了什么价值
3. **怎么体现**：引用具体代码或设计
4. **类比说明**：用日常生活中的类比帮助理解

## 阶段四：生成（Generate）— 结构化输出

### 文档模板

```markdown
# {项目名称} 项目详解

> 一句话定位 | 核心价值 | [快速跳转目录]

## 一、建立认知
### 1. 项目定位
### 2. 解决的问题与使用场景
### 3. 与同类项目的对比

## 二、理解设计
### 4. 核心理念与设计哲学
### 5. 核心概念模型
### 6. 整体架构

## 三、项目亮点
### 亮点 1：{标题}
### 亮点 2：{标题}
### ...

## 四、学会使用
### 7. 快速上手
### 8. 核心使用模式
### 9. 扩展与定制

## 五、深入掌握
### 10. 数据流与执行流
### 11. 可观测性与调试
### 12. 生态与集成

## 六、参与贡献
### 13. 开发指南
### 14. 技术实现要点

## 附录
- 关键文件索引
- 术语表
```

### 生成规则

1. **维度跳过**：不适用的维度标注"本项目无此机制"，绝不强行编造内容
2. **深度调节**：根据阶段一评估的项目规模，控制在目标长度范围内
3. **语言适配**：文档语言跟随用户请求语言
4. **示例必须**：每个核心使用模式必须包含代码示例
5. **溯源标注**：每个关键断言标注来源文件（如"参见 `schema/message.go`"），方便验证
6. **渐进披露**：每个章节从最简用法开始，逐步展示高级选项
7. **亮点突出**：亮点部分应让人感受到项目的魅力，而非干巴巴的技术罗列
8. **待验证标注**：未经源码确认的信息（如推断的功能、未读到的实现细节、第三方项目信息），在文档中用 `⚠️ 待确认` 行内标注，而非留到验证阶段才发现

## 阶段五：验证（Verify）— 回到源码核实

文档生成后，**必须回到项目源码进行交叉验证**，确保文档内容准确无误。

### 5.1 验证策略（按内容类型）

| 内容类型 | 验证方式 | 工具 |
|-------|--------|------|
| **代码示例** | 确认 API 调用、函数签名、参数名与源码一致；检查 import 路径、变量定义、类型匹配的完整性 | Grep 搜索签名 + Read 源码确认 |
| **接口/类型描述** | 确认方法、字段名、类型与源码一致 | Read 定义文件，逐字段比对 |
| **架构分层** | 确认包依赖方向与文档描述一致 | Grep 搜索 import 语句 |
| **流程描述** | 确认调用链与代码逻辑一致 | 沿调用链逐函数 Read |
| **配置/命令** | 确认构建命令、配置项与实际文件一致 | Read Makefile/CI 配置 |
| **同类对比** | 确认对比信息准确（避免编造） | WebSearch 查询验证 |
| **亮点描述** | 确认亮点所引用的代码确实存在且行为如描述 | Read 对应源码验证 |
| **版本/依赖** | 确认与依赖文件一致 | Read go.mod/package.json 等 |

### 5.2 验证优先级

| 优先级 | 内容 | 理由 |
|-------|------|------|
| **P0** | 文档中的代码示例（API 调用、函数签名、import 路径、变量完整性） | 用户最可能复制粘贴 |
| **P1** | 接口/类型定义 | 影响理解准确性 |
| **P2** | 架构和流程描述 | 影响整体认知 |
| **P3** | 配置和生态信息、亮点描述 | 影响操作准确性和印象 |

### 5.3 验证执行

1. 从生成的文档中提取所有可验证的断言
2. 优先确认文档中的 `⚠️ 待确认` 标注项
3. 按优先级排序（P0 → P3）
4. 逐项回到源码验证：
   - **P0**：Read 对应源文件，确认函数签名完全一致；检查代码示例中 import 路径是否正确、变量是否在使用前定义、类型是否匹配、**必需参数是否完整提供**
   - **P1**：Grep 确认类型/接口存在且字段正确
   - **P2**：沿描述的调用链追踪源码，确认流转正确
   - **P3**：Read 配置文件和亮点引用的源码确认准确
5. 记录所有发现的偏差
6. 修正文档中的错误
7. 确认或移除 `⚠️ 待确认` 标注
8. 将验证结果以日志形式输出到控制台（不写入文档），供用户了解文档质量

### 5.4 验证结果输出

验证完成后，将结果以控制台日志形式输出（**不写入文档**），格式如下：

```
📋 文档验证报告

✅ 验证通过（23 项）：
  - X 的描述 — 源码确认一致
  - Y 的描述 — 源码确认一致

🔧 已修正（2 项）：
  - Z 的描述（原文档有误，已修正为 ...）

⚠️ 待确认（1 项）：
  - W 的描述 — 需进一步确认
```

## 阶段六：精炼（Refine）— 提升可读性

验证完成后，进行轻量的精炼检查，提升文档的阅读体验。

### 精炼检查项

1. **重复表述**：检查是否有同一概念在不同章节重复解释，如有则保留最合适的那个，其他用简短引用替代
2. **冗长段落**：检查是否有超过 10 行的段落可以拆分或精简
3. **首尾呼应**：文档开头承诺的内容（如"五种使用模式"），后续是否兑现
4. **待确认清理**：所有 `⚠️ 待确认` 标注是否都已处理（确认后移除或保留为正式说明）
5. **长度合规**：文档总长度是否符合阶段一确定的目标长度范围

## 重要提醒

- 始终使用 Read/Grep/Glob 工具从源码验证信息，绝不凭推断
- 对不确定的断言，标注 `⚠️ 待确认` 而非猜测
- 对同类项目的对比，优先使用 WebSearch 验证事实；对比应客观公正，避免非黑即白
- 验证阶段不可跳过，每份生成的文档都必须经过验证
- 亮点发现是文档的差异化价值，不要写成技术罗列，要让人感受到项目的魅力
- 文档长度应与项目规模匹配，避免小型项目生成过度冗长的文档
